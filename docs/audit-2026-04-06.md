## 1. 架构（System Engineering）

### A1. ServiceRegistry 是包了一层皮的 Service Locator
**位置**：`ServiceRegistry.kt:24-93`

**问题**：
- 模块以接口暴露，但访问方式是 `ServiceRegistry.networkModule.xxx` 全局静态——任何类都能拿到任何模块，没有构造器注入、没有作用域限制。
- `clearAllCaches()`（88-93 行）硬编码三类缓存，每加一类要改这里——典型的横切泄漏。
- `initialize()` 没有运行时前置条件检查，"必须在 Settings/EhDB 之后调用"只靠注释。
- `LRRTagCache`（lrr 包外部单例）被 ServiceRegistry 直接引用，让 registry 反向依赖 client 子包。

**修复方向**：
1. 定义 `interface Cacheable { fun clearCache() }`，模块自注册到 registry 的清理总线。
2. `initialize()` 入参显式传 `Settings + EhDB`，缺一编译期就报错。
3. `LRRTagCache` 进 `ClientModule`，不要让单例与模块体系并存。

---

### A2. EhApplication 仍是隐性单例 + 副作用集散地
**位置**：`EhApplication.kt:54-237`

**问题**：383 行里至少做了 12 件事——locale wrap、UncaughtExceptionHandler、Crash、Settings、Auth、ReadableTime、AppConfig、EhDB、profile 加载、合库、Filter 加载、JNI、目录迁移、analytics、temp 清理、版本号写回。

`onCreate` 中两处用裸 `CoroutineScope(Dispatchers.IO)`（116, 129 行）启动孤儿协程，**因为 ServiceRegistry 还没准备好**——这暴露了**初始化顺序合约缺失**。

**修复方向**：
1. 抽 `AppInitializer` 聚合所有初始化步骤，并定义阶段（critical → bg-defer → post-display）。
2. 裸 scope 改成 `AppModule.bootScope`（带 SupervisorJob+CEH），ServiceRegistry 就绪后转交。

---

### A3. EhDB 双 API + Room/SQLiteOpenHelper 共存
**位置**：`EhDB.kt:97-285, 80-89`

**问题**：
- 同一个文件既维护 Room 的 Migration，又用 `SQLiteOpenHelper` 直读老 DB（`OldDBHelper`），还自己拿 InputStream 拷贝整个 db 文件做 export（`exportDB` 第 648 行）——三种数据库访问范式叠在一起。
- `blockingDb` 是已知技术债，文档明说"once those legacy paths have been migrated, this should be promoted to a hard throw"——说明这条路径躺了很久。

**修复方向**：
1. `mergeOldDB` 改为 `suspend fun`，去掉内部 `runBlocking`。
2. `exportDB` 走 Room 的 `db.openHelper.writableDatabase.path` 后再拷贝。
3. 全局 grep 列出所有 `EhDB.xxx()`（非 Async）调用方，分批迁移，然后 `blockingDb` 在 debug 改为 `error()`。

---

### A4. 已死的 EhViewer 残骸仍在网络栈里
**位置**：`module/ClientModule.kt:20`、`client/EhClient.kt`、`client/EhCookieStore.kt`

**问题**：
- `EhClient` 已被注释为 "E-Hentai engine methods removed"，但 `ClientModule` 仍 `by lazy { EhClient(context) }`。
- `EhCookieStore` 是 EH 的 cookie 仓库，被 OkHttp 当 `cookieJar` 注入（`NetworkModule.kt:50`）——但 LANraragi 用 Bearer token，根本不需要 cookie jar。

**修复方向**：删除 `EhClient`、`EhRequest`、`EhCookieStore`、`EhHosts`（如仅服务 EH），把 cookieJar 换成 `CookieJar.NO_COOKIES`。同时拔掉 `NetworkModule.kt:66-87` 那条 WebView 同步链。

---

### A5. Settings 双轨 + 残留 EH 国别策略
**位置**：`Settings.kt:60-73`，`Settings.kt`（444 行整体）

**问题**：`Settings.fixDefaultValue()` 检测到 `Locale.CN` 就开启 `BuiltInHosts/BuiltEXHosts/DomainFronting`——这是 E-Hentai 反封锁默认值，对 LRR 完全无意义且可能让用户误以为 LRR 也走这套 hosts。`Settings.kt`（legacy global @JvmStatic）和 `settings/*` 的模块化对象并行存在。

**修复方向**：
1. 立刻删 CN 国别分支。
2. 把 `Settings` 里剩下的字段切到对应的 modular settings，最终 `Settings.kt` 删成只剩 `getContext()/getPreferences()` 或彻底废弃。

---

### A6. AppModule 杂货铺
**位置**：`module/AppModule.kt:14-65`

**问题**：globalStuffMap（弱引用 ID 表）+ tempCacheMap（Intent 跨页传递）+ 4-thread `ExecutorService`（行 22）混在一起。grep 显示这个 `executorService` 几乎没人用——僵尸资源。

**修复方向**：拆 `ObjectHandleRegistry` / `IntentParcelStash`；删除 ExecutorService，调用方改用 `ServiceRegistry.coroutineModule.ioScope` 或 `Dispatchers.Default`。

---

### A7. Helper-via-Callback 模式扩散到不该用的地方
**位置**：`ui/scene/gallery/detail/Gallery{Tag,Download,DetailRequest}Helper.kt`、`ui/scene/download/Download{Label,Filter,Import}Helper.kt`

**问题**：把巨大的 Scene 拆成多个 Helper 是合理的；但每个 Helper 都通过 `Callback` 接口跟 Scene 双向耦合，状态散落，单元测试只能写假实现。`GalleryDetailViewModel` 已存在但被 Helper 旁路——形成"有 ViewModel 也用不到"的局面。

**修复方向**：把 Helper 当 ViewModel 内部 collaborator（构造时注入 ViewModel 暴露的 `MutableStateFlow`），删 Callback，让 Scene 只负责"订阅 state + 渲染"。

---

### A8. namespace 与 applicationId 名称错位
**位置**：全项目 `com.hippo.ehviewer.*` 包名 vs `com.lanraragi.reader` applicationId

**问题**：所有 SharedPreferences key、Provider authority、Crash log、内部日志 TAG 都还在 `ehviewer` 命名空间，给逆向、bug 上报、新成员阅读都增加了认知负担。

**修复方向**：分阶段重命名包到 `com.lanraragi.reader.*`，先迁 lrr 子树，再迁顶层。高风险，必须计划。

---

### A9. 数据类身份混乱
**位置**：`client/data/GalleryInfo.kt`

**问题**：`GalleryInfo` 同时是 Room Entity / Parcelable / API DTO / UI Model。`gid = SHA-256(arcid)` 派生、`token = arcid`、`category = -1` 占位——字段语义被 EH 与 LRR 双重含义污染。

**修复方向**：拆 `GalleryInfoEntity`(Room) / `LRRArchive`(已存在的 DTO) / `GalleryInfoUi`(UI) 三层，转换在 `mapper/` 包集中。

## 2. 安全（Security）

### S1. cleartext 全开 + 注释自相矛盾【CRITICAL】
**位置**：`res/xml/network_security_config.xml:11-37`

**事实**：
- 第 11-13 行注释说 "User-installed CAs are trusted"，第 22 行注释又说 "User-installed CAs are NOT trusted in release builds"。配置实际是 base 仅信任系统 CA，debug-overrides 信任 user CA——以 22 行那句为准，11 行那段是误导。
- `<base-config cleartextTrafficPermitted="true">` 让所有域名走明文都不报错。`LRRAuthInterceptor` 仅能保证不向他人发 token，但对**配置好的服务器本身**仍是 100% 明文 + 任何同 LAN 的人可直接抓包/篡改。
- 自签 HTTPS（reverse proxy 常见配置）反而进不来——release build 的 user CA 不信任。

**修复方向**：
1. base 改为 `cleartextTrafficPermitted="false"`。
2. 提供运行时白名单：用户配置服务器时勾选 "trust this LAN host (cleartext)"，APP 通过 OkHttp 拦截器拒绝非白名单 host 的明文请求。
3. 提供 "import server CA" 的设置项，把 user CA 添加到 OkHttp 的 `X509TrustManager`，解决自签问题。
4. 删除矛盾注释。

---

### S2. Interceptor 内同步 DNS + DNS Rebinding【CRITICAL】
**位置**：`client/lrr/LRRAuthInterceptor.kt:64-71, 38-41`

**威胁**：
- `InetAddress.getByName(reqHost)` 在网络拦截器中**同步**触发 DNS 解析。OkHttp 自身有 DNS，再做一层等于额外阻塞 OkHttp 工作线程。
- DNS 缓存被污染或攻击者控制 DNS（恶意 LAN 路由器/恶意 DoH），可让 `srvHost` 与 `reqHost` 短时间内解析到同一 IP，绕过 host 校验后续解析又指向攻击者——经典 DNS Rebinding。
- scheme 不参与比较：配置 `https://lrr.local:443` 时，攻击者构造 `http://lrr.local:443` 仍可通过校验，token 被以明文重发。

**修复方向**：
1. 不做 DNS 解析，**只比对字符串 host:port + scheme**。
2. IP literal 直接字符串等值比对（含 IPv6 normalize），不依赖 InetAddress。
3. scheme 必须严格相同（http vs https 视为不同 host）。
4. 升级一步：自定义 OkHttp `Dns`，把"非配置 host 解析为 0.0.0.0"，彻底不让请求出门。

---

### S3. 多处 URL 字符串拼接绕过 parseBaseUrl【HIGH】
**位置**：已确认 `client/lrr/LRRArchiveApi.kt:221`；agent 报告还指向 `LRRCategoryApi.kt:35,69`、`LRRDatabaseApi.kt:28,51,86,105`、`LRRMinionApi.kt:64`、`LRRServerApi.kt:26`（需要 grep 复核）

**威胁**：恶意 baseUrl `http://attacker.com#@your-server/` 解析行为依实现而异，可能把 token 发到 attacker.com。

**修复方向**：grep `"\$baseUrl/api"` 与 `"\${baseUrl}/api"`，全部改成 `parseBaseUrl(baseUrl).newBuilder().addPathSegments("api/...")`，并加 detekt rule 禁止字符串拼接 baseUrl。

---

### S4. EncryptedSharedPreferences 失败时 setter 静默【HIGH】
**位置**：`client/lrr/LRRAuthManager.kt:110-118, 129-132, 188-196, 225-251`

**问题**：所有 setter 都是 `val prefs = sPrefs ?: return`——KeyStore 挂掉时，用户改 server URL/api key/pattern 都默默丢弃，下次启动以为"忘了设置"。`sNeedsReauthentication` 只覆盖了**全局 key 路径**；多 profile 的 `setApiKeyForProfile` 没有任何兜底。

**修复方向**：
1. setter 改为返回 `Boolean` 或抛 `LRRSecureStorageUnavailableException`，UI 必须显式处理失败。
2. 在 ServerListScene、SetSecurityActivity 等所有写入入口加 catch + 用户提示。
3. KeyStore 故障的事实应**持久化到 plain prefs**，每次 setter 失败时强制刷新这个 flag。
4. `sNeedsReauthentication` 检查必须遍历所有 profile，不能只看 `KEY_SERVER_URL`。

---

### S5. PBKDF2 + 9 宫格 + 无尝试限制【HIGH】
**位置**：`client/lrr/LRRAuthManager.kt:224-277`

**威胁**：9 宫格手势空间约 38.9 万种（4-9 dot），PBKDF2-HMAC-SHA256 100K 迭代在桌面 GPU 上每秒可算约 10⁵ 次——离线穷举不到 1 小时。本地 5 次锁定在攻击者拿到 prefs 文件的场景下完全无效。

**修复方向**：
1. 与设备生物识别绑定：解锁手势校验前先 `BiometricPrompt`，校验通过后用 BiometricPrompt 提供的 `CryptoObject`（KeyStore-bound key）解开 hash。
2. 把迭代次数提到 200K-500K（缓解但不解决）。
3. 失败次数持久化到 plain prefs，跨重启计数。

---

### S6. WebView Cookie 同步——危险且没必要【HIGH】
**位置**：`module/NetworkModule.kt:66-87`

**事实**：所有响应（无论域名）的 `Set-Cookie` 都被写到 `WebView.CookieManager`。LANraragi 用 Bearer token，正常情况下根本没有 Set-Cookie。这段是 EhViewer 时代为了把 WebView 与登录态同步留下来的死代码。

**风险**：
- 一旦 LRR（或反向代理）下发 cookie，会被 APP 内任何 WebView 加载的页面读取。
- 拦截器把所有响应的 host 都写进去，等于跨域污染。

**修复方向**：直接删除整段拦截器，连同 EhCookieStore 一起清掉。

---

### S7. FileProvider 路径过宽【MEDIUM】
**位置**：`res/xml/filepaths.xml`（agent 报告含 `<cache-path path=""/>`、`<external-cache-path path=""/>` 等空 path）

**风险**：grantUriPermission 之后，第三方应用通过 `Intent.ACTION_VIEW`/`SEND` 收到的 URI 可访问整个 cache/external-cache 子树。

**修复方向**：把 path 收敛到具体子目录（`crop_temp/`、`share_temp/`），禁止空 path。

---

### S8. 日志泄漏 host【MEDIUM】
**位置**：`client/lrr/LRRAuthInterceptor.kt:39`

**问题**：`Log.w(TAG, "API key is being sent over plaintext HTTP to ${original.url.host}")`。proguard 是否真 strip `Log.w` 需验证。即使 strip 了，开发 build 也会进 logcat。

**修复方向**：去掉 host 输出；改为 `Log.e(TAG, "Refusing to send API key over plaintext HTTP")` 并真的拒绝发送（abort 这次请求）。

---

### S9. exportDB 路径校验缺失【MEDIUM】
**位置**：`EhDB.kt:648-667`

**问题**：`exportDB(context, file)` 没校验 `file` 是否落在受信任目录、不允许 follow symlink、不允许写 content://。恶意 Intent 可让用户把整库导出到任意位置。

**修复方向**：限制目标必须是 SAF 选出的 user-visible 目录，禁止 raw `File`。

## 3. 稳定性（Reliability）

### R1. DownloadManager 多处主线程独占的集合被 IO 写【CRITICAL】
**位置**：`download/DownloadManager.kt:68-69`（声明 main-thread only）

| 行号 | 调用上下文 | 写入的"主线程独占"集合 |
|---|---|---|
| 136-141 | `loadDataFromDb()` 在 IO 协程 | `mLabelList`、`mLabelSet`、`mMap` |
| 147 | 同上 | `mAllInfoList` |
| 164-178 | 同上 | `mAllInfoMap`、`mLabelList`、`mLabelSet`、`mMap` |
| 650-651 | `addDownload(List)` 内 `scope.launch{}` | `mLabelList`、`mLabelSet` |
| 680-684 | `addDownloadLabel()` 内 `scope.launch{}` | `mLabelList`、`mLabelSet` |
| 1076-1077 | `addLabel()` IO 段 | 修改主线程已添加对象的 `id`/`time` |

**为什么这不是工程洁癖**：
- `loadDataFromDb` 跑在 init 协程里，没有任何主线程 barrier。`assertMainThread()` 保护的所有 getter 只要在 init 完成前被调用，就是 HashMap 与 ArrayList 的真实并发访问，**会**抛 `ConcurrentModificationException` 或读到中间态。
- `addDownload(List)`、`addDownloadLabel` 是用户操作触发的——这些 IO 段与主线程后续操作完全并发。
- HashMap 在并发写下可能死循环（pre-7.0 是真死循环，8.0+ 是数据丢失）。

**修复方向**：
1. 所有"读写主线程独占集合"的工作必须在主线程完成。把 IO 段做成"先完成 DB 写、再 `Handler.post` 回主线程更新内存"的模式。
2. 加单测：构造 `loadDataFromDb` 仍在 IO 跑、主线程并发调用 `containLabel` 的场景，应能复现。

---

### R2. EhApplication 启动期裸协程 + 异常吞噬【HIGH】
**位置**：`EhApplication.kt:116-125, 129-133`

**问题**：
- 116、129 行用 `CoroutineScope(Dispatchers.IO).launch {}`，没有 SupervisorJob、没有 CEH。`getActiveProfileAsync()`/`mergeOldDB` 抛异常后**完全静默**。
- 第 116 行启动任务的下游："activeProfile 加载"和 ServiceRegistry 初始化（138 行）之间是**赛跑**——如果 profile 加载完成时 ServiceRegistry 还没初始化，后续路径会失败。

**修复方向**：
1. 增加 `AppModule.bootScope = CoroutineScope(SupervisorJob() + Dispatchers.IO + CEH)`，在 onCreate 第一行就创建。
2. 裸协程全部替换。
3. profile 加载结果改成 `Deferred<Long>` 让需要它的下游 await。

---

### R3. EhDB.blockingDb 主线程仅警告【HIGH】
**位置**：`EhDB.kt:80-89`

**问题**：注释承认"应在 debug 升为 hard throw"——但仍是 `Log.w`。这意味着 debug 也不抛，没有强迫机制驱动迁移。每个仍使用 `EhDB.xxx()`（非 Async）的调用方都是潜在 ANR。

**修复方向**：
1. 立刻在 debug build 改成 `error("EhDB main-thread blocking call: ${stack}")`。
2. CI 跑一次 instrumentation，列出所有 stack，开列迁移清单。

---

### R4. EhDB.mergeOldDB 内 runBlocking——不是 ANR，但仍多余
**位置**：`EhDB.kt:124, 134`

**澄清**：`mergeOldDB` 是从 `CoroutineScope(Dispatchers.IO).launch{}` 调用的，runBlocking 阻塞的是 IO 线程，**不会**触发 ANR。但 `runBlocking` 在协程内本身就是 anti-pattern，浪费一个 IO worker，且若 SQL 异常会被外层裸 scope 吃掉。

**修复方向**：把 `mergeOldDB` 改成 `suspend fun`，删掉 `runBlocking { ... }`。

---

### R5. EhFilter 异步加载导致短窗口"不过滤"【HIGH】
**位置**：`EhApplication.kt:143-149`

**问题**：filters 启动时为空，"安全默认 = 不过滤"。如果用户在加载完成前打开列表/搜索，应被屏蔽的内容会瞬间渲染出来。对**强制屏蔽未成年/敏感内容的用户而言，这本身就是安全问题**。

**修复方向**：
1. 在 `EhFilter.getInstance()` 上加 `awaitReady()` suspend，UI 在过滤前 await（可设 1s 超时，超时则放过 + 上报）。
2. 或干脆把 filter 加载放进 `EhApplication.onCreate()` 的关键路径（filter 表通常 < 100 项，启动延迟 < 50ms）。

---

### R6. WebView Cookie flush 上主线程【MEDIUM】
**位置**：`module/NetworkModule.kt:78-85`

**问题**：`Handler(Looper.getMainLooper()).postDelayed({ cookieManager.flush() }, 3000)`——`CookieManager.flush()` 第一次调用会**懒加载 WebView 进程**，主线程瞬间卡 100-500ms。

**修复方向**：参 S6 直接删整段拦截器；如果保留，把 flush 切到 background HandlerThread。

---

### R7. DownloadManager.awaitInit 用 runBlocking 套 withTimeout(10s)【HIGH】
**位置**：`download/DownloadManager.kt:198-206`

**问题**：从后台调用没问题，但 10s 是相当宽的硬上限——慢磁盘 + 大下载库（5000 行+）情况下能吃满。`DownloadService.onStartCommand` 若赶上首启冷加载，10s 很容易触发系统对前台 service 的 5s `startForeground` ANR。

**修复方向**：把它改成 `suspend fun`，调用方（DownloadService）改用 `lifecycleScope.launch`，先 `startForeground(占位 notification)` 再 await。

### R8. !! 滥用风险（方向正确，量级需复核）
**位置**：`client/lrr/data/*.kt`、`LRRArchiveApi.deleteArchive` 等的 `jsonPrimitive.content` 路径

**问题**：Kotlin 项目里大量 `!!` 通常是 Java 互操作字段，不一定是真危险。LRR API 的 JSON 解析路径有真实 NPE 风险（如果 LANraragi 升级删字段）。

**修复方向**：
- 用 Detekt 的 `UnsafeCallOnNullableType` 规则只对 `client/lrr/**` 与 `dao/**` 开启，逐步清零。
- 对 `client/lrr/data/*.kt` 的 `@Serializable` 类做契约测试（用旧 LANraragi 1.8、当前 1.9 的真实响应做 fixtures）。

### R9. Crash.saveCrashLog 在 Settings 之前可被调用【LOW】

## 4. 性能（Performance）

### P1. 孤儿 label 同步 N 次 DB roundtrip【HIGH】
**位置**：`download/DownloadManager.kt:172-175, 649`

**代价**：每个孤儿 label 一次 `await EhDB.addDownloadLabelAsync` ≈ 5-30 ms。20 个孤儿 = 100-600 ms，且在 init 路径上。

**修复方向**：先 `getAllDownloadLabelListAsync()` 拿全集，diff 出缺失集合，一次 `withTransaction { dao.insertAll(missing) }`。

---

### P2. addDownload(List) 全表 sort + 监听器全广播【HIGH】
**位置**：`download/DownloadManager.kt:637, 643, 660-662`

**代价**：n=1000 时 `Collections.sort` 主线程 ~5-15ms；`onReload()` 触发 RecyclerView 全量 rebind 而不是 DiffUtil。

**修复方向**：用插入排序 + DiffUtil 增量更新；或让 DAO 本身按 `time DESC` 查，内存里压根不要 re-sort。

---

### P3. ClientModule 内存缓存上限对中高端机过紧【MEDIUM】
**位置**：`module/ClientModule.kt:42-46`

**事实**：`maxMemory()/8` clamp 到 [20MB, 64MB]。8GB 旗舰也只给 64MB；列表快速滑动几乎必中 disk cache。

**修复方向**：分档：`<512MB → 16MB`、`<1GB → 32MB`、`<3GB → 80MB`、`>=3GB → 128MB`。

---

### P4. AppModule.executorService 是僵尸【LOW】
**位置**：`module/AppModule.kt:22`

**事实**：`Executors.newFixedThreadPool(4)` 持有 4 个永久线程 ≈ 4MB 栈 + 调度开销。grep 调用方应该几乎为零。

**修复方向**：grep 确认后删除；剩余调用方迁到 `Dispatchers.IO`/`Dispatchers.Default`。

---

### P5. Settings 热路径无缓存【MEDIUM】
**位置**：`Settings.kt` 全部 getter

**问题**：Reader 每帧拿 `getReadingDirection()`、`getPageScaling()` 直接打 SharedPreferences。SP 读虽然 mmap 后很快，但锁竞争 + 频繁 `try/catch ClassCastException` 在 60FPS 路径上累计可观。

**修复方向**：高频字段缓存在 `volatile` 字段，注册 `OnSharedPreferenceChangeListener` 同步刷新；或迁 `DataStore.preferences`。

---

### P6. CookieJar 拦截器无差别全响应处理【MEDIUM】
**位置**：`module/NetworkModule.kt:66-87`

**澄清**：debounce 是聚合不是延迟（agent 报告把这点说反了）。但每次响应仍要执行 `headers("Set-Cookie")` 解析 + 主线程 post——大量请求时 post 队列被打满。

**修复方向**：参 S6/R6 直接删拦截器。

---

### P7. 启动期下载目录扫描无幂等【MEDIUM】
**位置**：`EhApplication.kt:191-218`

**事实**：`if (oldBase.exists() && oldBase.isDirectory)` —— oldBase 不删除就每次启动都 listFiles，对几千个旧目录 = 上百 ms 文件系统调用。

**修复方向**：迁移完成后写 `Settings.putBoolean("download_migration_v1_done", true)`，下次直接 short-circuit。

---

### P8. Paging 3 配置过于激进【MEDIUM】
**位置**：`ui/scene/gallery/list/GalleryListViewModel.kt`（CLAUDE.md 标 pageSize=100, prefetchDistance=20）

**问题**：每页 100 项 + 提前 20 触发——内存里常驻 200+ 个 GalleryInfo + 缩略图。低端机 OOM 风险。

**修复方向**：pageSize=50, prefetchDistance=10；或按设备分级。

---

### P9. OkHttp 缩略图 cache 头可能无效【MEDIUM】
**位置**：`module/NetworkModule.kt:53-65`

**问题**：写入了 `max-age=3600, stale-while-revalidate=82800`，但 LRR 缩略图 URL 通常带 query——OkHttp 默认对 query 请求降权缓存。

**修复方向**：先 log 一次缩略图 URL 形态确认；改为在 PagingSource 拼 URL 时不带 query；或自己写缓存层（key 用 arcid）。

---

### P10. attachBaseContext 同步 SP 读【LOW】
**位置**：`EhApplication.kt:64-77`

**澄清**：SP 第一次冷读通常 2-15ms，不是 50ms 量级，但确实在主线程冷启动路径上。

**修复方向**：把 language 写到一个独立 plain prefs file（更小、更快加载），或缓存为 process 静态。

### P11. ProGuard log strip 验证【MEDIUM】
**位置**：`app/proguard-rules.pro`

**修复方向**：`-assumenosideeffects class android.util.Log { public static *** v(...); public static *** d(...); public static *** i(...); public static *** w(...); }`，并加 CI 检查 mapping.txt。

### W0-1 `fix/download-manager-thread-safety`

**状态**：✅ 已合入 main (`ec39646` Merge W0-1)

**背景**
`DownloadManager.kt` 在 68-69 行声明所有共享集合是"主线程独占"，并用 `assertMainThread()` 强制公共方法。但实际有多处违反，是潜在的随机崩溃源（HashMap 并发写、ConcurrentModificationException）。

**现状**
违反点（已亲自核对）：
- `loadDataFromDb()` 第 132-190 行，IO 协程内写 `mLabelList / mLabelSet / mAllInfoList / mMap / mAllInfoMap`。
- `addDownload(List<DownloadInfo>)` 第 647-657 行，`scope.launch{}` 内写 `mLabelList / mLabelSet`。
- `addDownloadLabel()` 第 678-685 行，同款问题。
- `addLabel()` 第 1064-1078 行，`scope.launch` 内修改主线程已添加对象的 `id/time` 字段（无 volatile/同步）。

**目标**
所有"主线程独占"集合的写操作必须发生在主线程；IO 段只做 DB 读写。

**验收标准**
1. 上述四处全部修复：IO 段先做 DB 操作 → `Handler.post {}` / `withContext(Dispatchers.Main)` → 在主线程更新内存集合。
2. 所有公共 getter（`containLabel`/`labelList`/`getLabelCount` 等）增加 `assertMainThread()`。
3. 新增单元测试：构造 `loadDataFromDb` 在 IO 跑、主线程并发调用 `containLabel` 的场景，旧实现会复现 race，新实现稳定通过 1000 次。
4. `./gradlew app:testAppReleaseDebugUnitTest` 全绿。
5. 没有任何新的 `runBlocking`、`Thread.sleep`。

### W0-2 `security/lrr-auth-interceptor-hardening`

**状态**：✅ 已合入 main (`9f1037c` Merge W0-2)

**背景**
`LRRAuthInterceptor` 是防止 token 泄漏给非配置主机的最后防线，但当前实现存在 DNS Rebinding 绕过 + 阻塞 OkHttp 工作线程 + scheme 不校验三个问题。

**现状**
- `client/lrr/LRRAuthInterceptor.kt:64-71`：`InetAddress.getByName()` 同步 DNS，且仅按首条 A 记录比对。
- 第 38-41 行：明文 HTTP 时只 `Log.w` 不 abort，仍把 token 发出去。
- 没有比较 scheme（http vs https 同 host:port 视为同一个）。
- 第 39 行 `Log.w` 日志输出 host，可能在 release 未被 strip。

**目标**
- 校验逻辑变成纯字符串等值比对（host + port + scheme），不做任何 DNS 解析。
- 配置主机为 `http://*` 且请求是同一主机时按现状放行；任何 https→http 或 http→https 不一致都拒绝。
- 明文路径直接 abort 请求并抛 `LRRPlaintextRefusedException`，不再 Log + 继续。
- 日志去掉 host。

**验收标准**
1. 拦截器内不再调用 `InetAddress`、不再调用 `Dns`、不再有任何 IO。
2. 新增单测：覆盖 IPv4/IPv6 字面量、IPv6 mapped、scheme mismatch、port default vs explicit、userInfo `@`、host case 等所有边界。
3. 单测断言：明文 HTTP 配置时，正常请求放行；HTTPS 配置但请求是 HTTP，拒绝。
4. `Log.w` 改为 `Log.e`，输出不含 host。
5. 手动验证：恶意 baseUrl `http://attacker.com#@configured.local/api/foo` 被拒绝。

### W0-3 `security/network-config-tightening`

### 基底偏移问题

所有 worktree 的实际基底是 `da8d93e`（版本号提升 commit），而非预期的 `fa74a62`（main HEAD）。中间隔了两个用户临时插入的 commit（`69ee4c0` fix + `fa74a62` feat）。这两个 commit 改的文件（`GalleryInfo.kt`、`LRRArchive.kt`、`GalleryAdapter*.kt`、`GalleryListScene.kt`、`GalleryDetailScene.kt`、`GalleryActivity.kt` 等）与 W2 的 9 个分支零交集，因此 cherry-pick 全部 clean apply，无需手动解决冲突。

W2-4 的 agent 自行发现基底问题并执行了 `git rebase main`；其余 agent 未注意到。主会话统一采用 cherry-pick 策略。

### 经验教训（session 级）

1. **worktree 基底不可靠**：`isolation: "worktree"` 的基底取决于 agent 启动时的 detached HEAD 状态，实际可能落后于主会话的 main。对策：主会话合入时一律用 `git cherry-pick <task-commit>` 而非 `git merge <branch>`，避免引入 worktree 缺失 commit 的反向差异。
2. **先查 `git diff-tree --name-only` 再 cherry-pick**：确认 task commit 只改了预期文件，不含基底差异污染。
3. **W2-4 / W2-5 同文件不同区域的并行安全**：git 三路 merge 在 W2-4 cherry-pick 后、W2-5 cherry-pick 时自动 auto-merge `GalleryListViewModel.kt`（accessor 删除 vs companion constant 修改），零冲突。
4. **DiffUtil 替换 notifyDataSetChanged 的历史教训**：W2-2 原计划包含 listener 重构→DiffUtil，但 memory 记录（`feedback_notifydatasetchanged.md`）标注上次尝试导致崩溃已回退。本轮 scope 缩减为仅 sort 优化，DiffUtil 迁移需要独立调查根因后再尝试。
5. **调研类任务可能降级为 doc-only**：W2-8 原计划修复 thumbnail cache key 问题，但调研发现 URL 无 query param、cache 本身有效，最终只补了注释。这是合理的——审查报告基于"可能"的假设，落地时验证假设是必要步骤。

## 附录 F：W3-7 Stage 2/3 可行性评估（2026-04-11）

> 结论：**不推荐执行。** Stage 1 已迁移高频编辑的 API 客户端（25 文件），ROI 合理。Stage 2/3 的 ~300 文件迁移收益纯 cosmetic，风险不可接受。

### 耦合现状

`com.hippo.*` 框架（230 Java 文件，标注为 "stable legacy, rarely touched"）对 `com.hippo.ehviewer.*` 存在 **38 个反向依赖文件**：

| 耦合类型 | 数量 | 详情 |
|---------|------|------|
| framework → ehviewer.R | 32 Java 文件 | 框架 widget/preference/scene 直接引用 R 资源 |
| framework → ehviewer 业务类 | 7 Java 文件 | Settings(2), UrlOpener(2), EhApplication(3), ServiceRegistry(3), EhActivity(1), PermissionCallBack(1) |
| XML layout 自定义 View FQN | 24 布局文件, 27 处引用 | `<com.hippo.ehviewer.widget.SearchBar>` 等 |
| AndroidManifest 组件声明 | 9 处 | Application + 7 Activity + 1 Service |
| SharedPreferences 文件名 | 1 处 | `PreferenceManager.getDefaultSharedPreferences()` 文件名 = `{namespace}_preferences` |
| gradle namespace | `com.hippo.ehviewer` | 决定 R 类包名、BuildConfig 包名、SP 默认文件名 |

### 评估的三个方案

**方案 A：仅移源码，不改 namespace**
- 移 ~100 个非 UI 文件到 `com.lanraragi.reader.*`
- 不改 gradle namespace → R/SP 不受影响
- 代价：每个移动的文件需加 `import com.hippo.ehviewer.R`（视觉噪音）；更新 7 个框架 import
- 问题：两个命名空间混用反而更乱，迁移只做一半

**方案 B：改 namespace + 全量迁移**
- 改 gradle namespace → `com.lanraragi.reader`
- 移 ~200 个 Kotlin 文件 + 更新 32 个框架 Java R import + 24 个 XML 布局 + 9 个 Manifest 声明
- 必须写 SP 数据迁移（旧文件名 → 新文件名，失败 = 用户设置全丢）
- XML FQN 漏改 → 运行时 `ClassNotFoundException`（编译不报错）

**方案 C：到此为止（推荐）**
- Stage 1 已迁移最有价值的部分（LANraragi API 客户端，日常开发最频繁接触）
- `applicationId` 已经是 `com.lanraragi.reader`，用户可见的包名已正确
- 剩余 `com.hippo.ehviewer` 代码功能稳定、改动频率低
- 300 文件 + SP 迁移 + XML 改动的风险远大于 cosmetic 收益
