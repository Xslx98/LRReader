## 1. 稳定性（Reliability）

### R1. retryOnFailure 尾部 `lastException!!` 可能 NPE【HIGH】
**位置**：`client/api/LRRApiUtils.kt:152`

**事实**：
```kotlin
var lastException: Exception? = null
repeat(maxRetries + 1) { attempt ->
    try { return block() }
    catch (e: IOException) { lastException = e; ... }
}
throw lastException!!  // ← block() 若不抛 IOException 但抛其他异常？
```

`block()` 只 catch `IOException`。若 `block()` 抛出 `CancellationException`、`IllegalStateException` 等**非 IOException**，该异常不会被捕获，直接 propagate——这条路径是安全的。但控制流分析表明 `repeat` 正常退出（即所有 `maxRetries+1` 次尝试都成功执行了 `return block()`）不可能到达第 152 行，因为 `return` 已经退出函数。所以 `lastException!!` 实际只在 `catch` 执行过后才会到达，此时 `lastException` 必定非 null。

**真正的风险**是编译器/静态分析无法证明这一点——未来添加新的 catch 分支或修改控制流时容易引入 NPE。

**修复方向**：改为 `throw lastException ?: IOException("Retry exhausted without specific exception")`，消除 `!!` 的脆弱性。

### R2. LRRTagCache.suggest() volatile 快照不一致【MEDIUM】
**位置**：`client/api/LRRTagCache.kt:88-94`

**问题**：
```kotlin
val currentTags = tags        // volatile read #1
val currentKeys = searchKeys  // volatile read #2  ← 可能是不同版本！
for (i in currentTags.indices) {
    if (i < currentKeys.size && currentKeys[i].contains(lower)) {
```

`tags` 和 `searchKeys` 是两个独立的 `@Volatile` 字段。在 `getTags()` 的 mutex 保护下它们是**先写 searchKeys 再写 tags**（第 52-53 行）。`suggest()` 读取顺序是**先读 tags 再读 searchKeys**。极端情况下可能读到新 tags + 旧 searchKeys，导致 `currentKeys[i]` 越界或匹配错误的 key。

**实际影响**：`i < currentKeys.size` 的保护避免了 IOOBE，但匹配结果可能包含错位的建议。对搜索建议功能是 cosmetic 错误，不会崩溃。

**修复方向**：将 `tags` 和 `searchKeys` 合并为一个不可变数据类 `CacheSnapshot(tags, searchKeys)`，单个 `@Volatile` 字段赋值保证原子性。

### R3. LRRTagCache.suggest() 返回 subList 视图【LOW】
**位置**：`client/api/LRRTagCache.kt:100`

**问题**：`results.subList(0, limit)` 返回的是 `results` 的 backed view，而非独立副本。若调用方持有引用期间 `results` 被 GC 前修改（当前代码中不会，因为 `results` 是局部变量），理论上存在 `ConcurrentModificationException` 风险。

**修复方向**：改为 `results.take(limit)`。

### R4. Listener 解绑无保障——前次 R10 遗留【MEDIUM】
**位置**：`download/DownloadManager.kt:71` (`mDownloadInfoListeners`)

**问题**：Scene 销毁路径复杂（popScene/Activity finish/low memory kill），任何分支漏掉 `removeDownloadInfoListener` 都让 DownloadManager 永久强引用 Scene。GalleryListScene、DownloadsScene 都在 `onCreateView3` 注册 listener，在 `onDestroyView` 解绑——但 low memory kill 不保证走 `onDestroyView`。

**当前 37 处 `SimpleHandler` 引用中有多处在 listener 回调路径上**，意味着泄漏的 listener 还会通过 `SimpleHandler.getInstance().post()` 向已销毁的 View 发消息。

**修复方向**：
1. `mDownloadInfoListeners` 改为 `WeakHashMap<DownloadInfoListener, Unit>` 或 `CopyOnWriteArrayList<WeakReference<DownloadInfoListener>>`，每次回调前清理已 GC 的引用。
2. 或利用 `viewLifecycleOwner.lifecycleScope` 在 `ON_DESTROY` 自动移除。

## 2. 安全（Security）

### S1. SecurityScene 使用已废弃 FingerprintManager【MEDIUM】
**位置**：`ui/scene/SecurityScene.kt:103, 131-132`、`ui/SetSecurityActivity.kt:75`

**问题**：
```kotlin
mFingerprintManager = context.getSystemService(FingerprintManager::class.java)
mFingerprintManager!!.authenticate(null, mFingerprintCancellationSignal, 0, ...)
```

`FingerprintManager` 从 API 28 开始被标记为 `@Deprecated`，Google 推荐使用 `androidx.biometric.BiometricPrompt`。当前代码的问题：

1. **不处理 Class 3 / Class 2 / Class 1 生物识别分级**——`FingerprintManager` 只识别指纹，不支持 Face、Iris 等 Class 3 传感器。
2. `mFingerprintManager!!.authenticate(null, ...)` 的第一个参数 `crypto: CryptoObject?` 传 null，**没有与 W3-6 的 KeyStore-bound AES-GCM 集成**——即 W3-6 的 `BiometricPrompt` + `CryptoObject` 路径在 pattern lock 侧已实现，但指纹解锁侧仍走旧的 raw FingerprintManager，KeyStore 绑定被绕过。
3. 无 `@RequiresPermission(USE_BIOMETRIC)` 检查。

**修复方向**：替换为 `androidx.biometric.BiometricPrompt`（依赖已在 `libs.versions.toml` 中：`biometric = "1.1.0"`），统一 pattern lock 和指纹解锁的 KeyStore-bound 流程。

### S2. CookieDatabase / CookieRepository 死代码残留【LOW】
**位置**：`com/hippo/network/CookieDatabase.java`、`com/hippo/network/CookieRepository.java`

**问题**：W0-6 删除了 `EhCookieStore` 并把 OkHttp 的 `CookieJar` 换成了 `NO_COOKIES`，但底层的 `CookieDatabase`（非加密 SQLite）和 `CookieRepository` 仍然存在于 `com.hippo.network` 包中。虽然不再被运行时调用，但：
- 代码维护者可能误以为 cookie 仍在使用
- 非加密数据库文件可能仍存在于旧安装的设备上

**修复方向**：删除 `CookieDatabase.java` 和 `CookieRepository.java`。

**位置**：多处

## 3. 性能（Performance）

### P1. notifyDataSetChanged 大面积使用——前次 P2 的 DiffUtil 替换遗留【HIGH】
**位置**：全项目约 25 处（见下表）

| 文件 | 行号 | 场景 |
|---|---|---|
| `ui/scene/gallery/list/GalleryListScene.kt` | 202, 207, 208, 211, 218, 486 | DownloadInfoListener 各回调 |
| `ui/scene/gallery/list/GalleryAdapter.kt` | 113, 130 | LIST↔GRID 切换 |
| `ui/scene/gallery/list/GalleryAdapterNew.kt` | 118, 135 | 同上 |
| `ui/scene/download/DownloadsScene.kt` | 283, 629 | 数据刷新 |
| `ui/scene/download/part/MyPageChangeListener.kt` | 71 | 分页切换 |
| `ui/scene/gallery/list/GalleryDrawerHelper.kt` | 188 | 筛选器变更 |
| `ui/scene/history/HistoryScene.kt` | 194 | 历史清空 |
| `ui/scene/gallery/list/QuickSearchScene.kt` | 72, 182 | 快速搜索列表 |
| `widget/SearchBar.kt` | 230 | 搜索建议 |
| `ui/scene/gallery/detail/TagEditDialog.kt` | 384 | Tag 编辑 |

**问题**：`notifyDataSetChanged()` 会使 RecyclerView 放弃所有 ViewHolder 并全量 rebind，在画廊列表 50+ 项 + 缩略图场景下：
- 主线程帧时间飙升（16ms 以上 = 掉帧）
- 滚动位置丢失
- 缩略图重新加载闪烁

**历史**：前次 W2-2 原计划将 listener 广播改为 DiffUtil，但因**历史崩溃记录**（见 `memory/feedback_notifydatasetchanged.md`）缩减 scope。崩溃根因至今未定位。

**修复方向**：
1. **先定位 DiffUtil 崩溃根因**：复现条件（推测是 DiffUtil callback 执行期间数据源被修改——W0-1 的线程安全修复可能已消除此根因）。
2. 对 `GalleryListScene` 的 `DownloadInfoListener` 回调：大多数事件（`onAdd`、`onRemove`、`onChange`）只影响 1 个 item，应改为 `notifyItemChanged(position)` / `notifyItemInserted(position)` / `notifyItemRemoved(position)`。
3. LIST↔GRID 切换（`GalleryAdapter.setType`）：重建 LayoutManager 即可，不需要 `notifyDataSetChanged()`。

## 4. 架构（Architecture）

### A1. DownloadManager 仍是 God Class（1452 行）【CRITICAL】
**位置**：`download/DownloadManager.kt`

**问题**：虽然 W0-W3 修复了线程安全、孤儿 label 批量化、sort 优化等具体 bug，但类本身的**职责过载**未改变：

|---|---|
| 8 个共享集合管理 (mAllInfoList, mAllInfoMap, mMap, mLabelCountMap, mLabelList, mLabelSet, mDefaultInfoList, mWaitList) | ~100 |

| Worker 生命周期管理 (mActiveTasks, mActiveWorkers) | ~150 |
| Listener 分发 (DownloadInfoListener × 12 events, DownloadListener × 5 events) | ~200 |
| Speed tracking delegate | ~30 |

**影响**：
- 单元测试需要 mock 庞大的前置状态——`DownloadManagerTest` 已 29 个测试但仍只覆盖 happy path
- 任何修改都有波及全类的风险
- DownloadsViewModel 本应独立于 DownloadManager，但实际只是薄代理

**修复方向**：拆分为：
- `DownloadRepository`：集合管理 + DB 持久化
- `DownloadScheduler`：Worker 调度 + 状态机
- `DownloadLabelManager`：Label CRUD
- `DownloadEventBus`：Listener 分发（或替换为 SharedFlow）

### A3. DownloadService 反向依赖 UI 层【HIGH】
**位置**：`download/DownloadService.kt:39-41`

```kotlin
import com.hippo.ehviewer.ui.MainActivity
import com.hippo.ehviewer.ui.scene.download.DownloadsScene
```

**问题**：Service（后台服务层）直接 import Activity 和 Scene——这是经典的**反向依赖**。Service 构造 PendingIntent 时直接引用 `MainActivity.class` 和 `DownloadsScene::class.java` 用于通知栏点击跳转。

**影响**：
- Service 层无法独立于 UI 层测试
- UI 类重命名/移动时 Service 编译失败
- 违反分层原则：data/service → domain → UI

**修复方向**：
1. 通知栏跳转改为 Intent action string（如 `ACTION_SHOW_DOWNLOADS`），由 MainActivity 的 intent filter 路由
2. 或将 PendingIntent 构造逻辑移到 UI 层，Service 通过接口/回调获取

### A4. SimpleHandler 非线程安全单例——37 处引用【HIGH】
**位置**：`com/hippo/lib/yorozuya/SimpleHandler.java:30-35`

```java
public static Handler getInstance() {
    if (sInstance == null) {                    // ← 无同步
        sInstance = new Handler(Looper.getMainLooper());
    }
    return sInstance;
}
```

**问题**：
1. **非线程安全的 lazy init**：多线程首次调用可能创建多个 Handler 实例（虽然不会崩溃，但违反单例契约）
2. **全局 mutable 状态**：37 处引用遍布 `DownloadManager`(8)、`GalleryActivity`(8)、`GallerySliderController`(6)、`DownloadService`(4)、`DownloadSpeedTracker`(4)、`EhApplication`(2)、`MainActivity`(2)、`GalleryDetailScene`(2) 等
3. **隐性依赖**：所有依赖 `SimpleHandler` 的类无法在没有 Android Looper 的环境下单元测试
4. **与协程重复**：大多数 `SimpleHandler.getInstance().post { ... }` 等价于 `withContext(Dispatchers.Main) { ... }`

**修复方向**：
1. 短期：加 `synchronized` 或改为 `object` 单例 with `@JvmStatic`
2. 长期：逐步替换为 `withContext(Dispatchers.Main)` 或 ViewModel 的 `viewModelScope`

### 经验教训（session 级）

1. **Bash 权限被拒已从"偶发"升级为"系统性"**：W0-W3 期间约 30% agent 被拒，W4 达到 100%。Agent prompt 必须假设 Bash 不可用——prompt 的核心价值转移到"精确描述需要做什么改动"而非"让 agent 自主完成全流程"。
2. **Cherry-pick 仍然是最安全的合入策略**：W4-1/2/3 的 worktree 基底正确（`106da15`），cherry-pick clean apply 无冲突。
3. **W4-5 暴露了删除主代码时必须 grep 测试目录的教训**：删除 `CookieDatabase.java` + `CookieRepository.java` 后首次 build 失败——`CookieRepositoryTest.java` 引用了被删类。追加删除测试文件后全绿。未来的 `chore/remove-*` 分支 prompt 应显式要求"grep 测试目录中的引用"。
4. **W4-6 Fix 3（Thread → ByteArrayOutputStream）保留了一次 Thread**：`entry.extract()` 是同步阻塞 API（7-Zip JNI），无法被协程 cancel，也无法在 IO dispatcher 上直接执行（因为它写到 `Pipe.outputStream`）。最终方案是保留一次 Thread 做 extract → ByteArrayOutputStream，然后用 `decodeByteArray` 两次（bounds + real），消除了第二次 Thread + Pipe。完全消除 Thread 需要 A7Zip 库提供 `suspend` 或 `InputStream` 接口——不在本分支 scope 内。

### W5-3 fixup 详情

Agent 完成了 12 个文件的编辑，但 `DownloadManager.kt` 的 listener 循环替换不完整（6/18），原因是 `forEachListener {}` 替换了 `for (l in ...)` 但循环体内的 `l.onXxx()` 只在特定缩进模式下被替换为 `it.onXxx()`。主会话先用 `sed` 全局替换 `for (l in mDownloadInfoListeners) {` → `forEachListener {`，再手动修复 4 处残留的 `l.` → `it.`。另修复 `SpiderDen.kt` 的 `@JvmField` + smart cast 问题。

### 经验教训

1. **Agent 的 sed/replace 模式匹配不可靠**：对不同缩进级别的相同语义代码，agent 只处理了部分匹配。主会话应在 cherry-pick 后立即 build 验证，而非信任 agent 的"全部完成"声明。

3. **CompletableFuture 是 suspend→Java 接口的合理桥接**：`ThumbDataContainer` 实现 Java `DataContainer` 接口，方法不能 `suspend`。`CompletableFuture` + eager coroutine 构造是正确方案，不引入 `runBlocking`。
